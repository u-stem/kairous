# Issue 管理ガイド

## タイトル命名規則

Issue タイトルは **何を解決/実現するか** を端的に示す。Conventional Commits の prefix (`fix:`, `feat:` 等) は付けない (あれは Git コミット専用)。

| 区分 | 書き方 | 良い例 | 悪い例 |
|------|--------|--------|--------|
| エピック | テーマ / カテゴリ名 | `セッション機能` `パフォーマンス最適化` | `feat: セッション機能の実装` |
| PBI | 解決する課題 or 提供する価値 | `total_cards と daily_logs の race condition 修正` | `fix: race condition を直す` |
| サブタスク | 具体的な作業内容 | `increment_total_cards RPC 関数の作成` | `feat: RPC を追加` |

### ルール

- 日本語を基本とし、技術用語 (race condition, RPC 等) はそのまま使う
- 50文字以内を目安にする
- 「〜の実装」「〜の修正」「〜の追加」等、動作を示す名詞で終える
- 内容ラベルと重複する情報は省略可 (「バグ」ラベルがあれば「不具合:」は不要)

## ラベル

Issue 作成時に **区分** と **内容** の2軸でラベルを付与する。

### 区分ラベル (必須、1つ)

Issue の階層を示す。一覧で一目でわかるようにする。

| ラベル | 色 | 用途 |
|---|---|---|
| エピック | 紫 (濃) | 大テーマ。サブタスクを持つ親 Issue |
| PBI | 紫 (中) | プロダクトバックログアイテム。エピック配下の実装単位 |
| サブタスク | 紫 (薄) | PBI またはエピックの子タスク |

### 内容ラベル (必須、1つ以上)

Issue の性質を示す。

| ラベル | 用途 | 例 |
|---|---|---|
| バグ | 不具合の報告・修正 | 「カード追加後に total_cards が更新されない」 |
| 機能 | 新機能の追加 | 「セッション実行画面の実装」 |
| 改善 | 既存機能の改善 (UX, リファクタ, パフォーマンス) | 「getCard 単体取得アクション追加」 |
| 基盤 | CI/CD, ツール, 依存関係, インフラ | 「Node.js 24 対応」 |
| ドキュメント | ドキュメントの追加・修正 | 「API 仕様書の追加」 |
| 対応しない | 対応しないと判断したもの | 理由を記載して close |
| 破壊的変更 | DB マイグレーション等、注意が必要な変更 | 「sessions テーブルのスキーマ変更」 |

### 付与例

- `#14 リファクタ / 技術的負債` → `エピック` + `改善`
- `#12 total_cards race condition` → `サブタスク` + `バグ`
- `セッション機能の実装` → `PBI` + `機能`
- `DB スキーマ変更を伴う新機能` → `PBI` + `機能` + `破壊的変更`

### 判断基準

- 迷ったら内容は「改善」を付ける (機能 vs 改善で迷う場合)
- 「破壊的変更」「対応しない」は他の内容ラベルと併用可

## トリアージ

Issue は作成後、必ず以下のいずれかに振り分ける。

1. **即対応**: 重大バグ・セキュリティ問題。現在の MS に追加し最優先で着手
2. **PBI 化**: エピック配下の PBI として登録し、適切な MS に紐付ける
3. **対応しない**: 「対応しない」ラベルを付け、理由を記載して close

「Issue のまま open で放置」は禁止。

## 常駐エピック

| エピック | Issue | サブタスクの例 |
|---|---|---|
| リファクタ / 技術的負債 | #14 | コード品質改善、設計見直し |
| セキュリティ / 依存関係更新 | #15 | 脆弱性対応、パッケージ更新 |
| パフォーマンス最適化 | #16 | クエリ最適化、バンドルサイズ削減 |

### マイルストーンとの関係

- 常駐エピック本体にはマイルストーンを設定しない (永続的なカテゴリのため)
- 配下の具体タスク (サブタスク/PBI) は着手時に該当マイルストーンに紐付ける
- 「マイルストーンなし = 管理漏れ」ではなく、常駐エピック本体のみが例外

## マイルストーン

マイルストーン = リリース単位。テーマを設定するが、機能・バグ・基盤を排他せず同居させる。

### バージョニング (SemVer)

`vMAJOR.MINOR.PATCH` で管理する。v1.0.0 未満は開発フェーズ。

| バージョン | 条件 | 例 |
|-----------|------|-----|
| PATCH (v0.X.Y) | 重大バグ・セキュリティの緊急修正。機能追加なし | v0.3.1 認可バイパス修正 |
| MINOR (v0.X.0) | 機能追加リリース。通常のマイルストーン | v0.4.0 Stats 可視化 |
| MAJOR (vX.0.0) | 破壊的変更を含むリリース、またはプロダクション公開 | v1.0.0 MVP |

- **v0.X の間**: DB スキーマ・API の破壊的変更は MINOR で許容する (「破壊的変更」ラベルで追跡)
- **v1.0.0 の条件**: 全学習手法 (SRS, Active Recall, Interleaving, Elaboration, Pomodoro, Wakeful Rest, Free Study) が実装され、ユーザーが日常利用可能な状態
- **パッチリリース**: 通常の MS サイクルを待たず、修正 -> レビュー -> main マージ -> タグ作成で即リリース

### ルール

| ルール | 内容 |
|--------|------|
| 定義 | リリース単位 (vX.Y.Z)。テーマはあるが排他ではない |
| バグ | 重大: 現 MS に即追加。軽微: 次 MS |
| 基盤 | 開発効率に直結: 現 MS。それ以外: 次 MS |
| 完了条件 | MS 内の全 Issue が closed |
| スコープ管理 | 膨らんだら Issue を次 MS に移動。MS の肥大化を防ぐ |

### 作業順序

MS 内の Issue は以下の優先度で着手する。

1. **重大バグ** -- データ不整合、セキュリティ問題
2. **基盤** -- CI/テスト/Review など開発効率に直結するもの
3. **機能** -- MS のテーマとなる新機能
4. **改善** -- パフォーマンス、リファクタ

| MS | テーマ | 状態 |

## Issue テンプレート

`.github/ISSUE_TEMPLATE/` に配置済み。新規 Issue は必ずテンプレートを使用する。

| テンプレート | 用途 |
|---|---|
| `pbi.yml` | PBI (プロダクトバックログアイテム)。**触るファイル領域** 必須 |
| `bug.md` | バグ報告 |
| `feature.md` | 機能リクエスト |

`pbi.yml` のみ [Issue Forms](https://docs.github.com/issues/tracking-your-work-with-issues/using-issue-forms) 形式 (必須項目を強制するため)。bug/feature は Markdown 形式。

## マルチセッション協調

独立した Claude Code セッションが並行稼働する場合、ファイル競合を避けるため以下のルールに従う。詳細は [`.claude/rules/workflow.md` の「マルチセッション協調」節](.claude/rules/workflow.md) を参照。

### 基本方針

- 通信は GitHub Issue + コメント + Project Status のみを用いる (Claude Code の Agent Teams は同一親セッション内のみサポート)
- PBI には必ず「触るファイル領域」を記載する (`pbi.yml` テンプレート必須項目)
- 着手時に Issue コメントで宣言する: `着手: セッション α / 開始 日時 / 領域: ...`

### 着手前の確認フロー

1. `gh issue list --repo u-stem/kairous --state open --json number,title,projectItems` で他セッションの In Progress / In Review Issue を抽出
2. 自分の PBI のファイル領域と重複しないか目視確認
3. 重複する場合は:
   - 先行 Issue の完了を待つ (blocked by で明示)
   - 自分の範囲を狭める
   - 領域の切り分けを Issue コメントで合意する

### Project Status の自動遷移

手動で Project の Status を動かす必要はほぼない (Epic #196 で自動化済み)。PR の open / draft 切替に連動する。手動操作は「ブロッカー発生時の状態巻き戻し」など例外ケースのみ。

## Discussions の使い分け

Issue は具体的なタスク、Discussions はそれ以外の議論・記録に使う。

| カテゴリ | 用途 | 例 |
|---|---|---|
| アナウンス | リリース、マイルストーン完了の告知 | 「v0.2.0 リリース」 |
| 設計判断 (ADR) | なぜ X ではなく Y を選んだかの記録 | 「base-nova 採用の経緯」 |
| 振り返り | エピック/マイルストーン完了後のレトロスペクティブ | 「v0.2.0 Materials 振り返り」 |
| 運用ルール | 開発フロー、規約の議論 | 「レビューコメントルールの提案」 |
| アイデア | まだ Issue 化するほど具体的でない構想 | 「モバイルアプリ対応の可能性」 |

### Issue vs Discussion の判断基準

- コードの変更が必要 → Issue
- 議論・記録・アイデア段階 → Discussion
- Discussion で具体化されたら Issue に変換
