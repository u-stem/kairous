# CI 高速化 実装計画 (#231)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Next.js production build を CI 実行あたり 1 回に正規化し、Playwright E2E を shard 並列化することで、Lighthouse CI と test-large の所要時間を短縮する。

**Architecture:** build ジョブを新設して `.next/` を upload-artifact で共有。lighthouse / test-large は download-artifact で再利用。test-large は `playwright --shard` + matrix で 4 並列化。共通 setup は `.github/actions/setup-kairous` composite action に集約。

**Tech Stack:** GitHub Actions, actions/cache, actions/upload-artifact, actions/download-artifact, Playwright --shard, Next.js incremental build cache

**Spec:** [docs/superpowers/specs/2026-04-14-ci-speedup-design.md](../specs/2026-04-14-ci-speedup-design.md)

## Task 3: build ジョブ新設 + .next/cache キャッシュ + artifact upload

**Files:**
- Modify: `.github/workflows/ci.yml`

- [ ] **Step 1: actions/cache の SHA を解決**

```bash
gh api repos/actions/cache/git/refs/tags/v4.2.3 --jq .object.sha
```

取得した SHA を以下の YAML の `<cache-sha>` に埋める (無ければ `v4.2.2` でも可、同様に解決)。

- [ ] **Step 2: build ジョブを ci.yml に追加**

ci.yml の `test-medium` の後、`test-large` の前に挿入:

```yaml
  build:
    runs-on: ubuntu-latest
    needs: [lint-and-typecheck]
    steps:
      - uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
      - uses: ./.github/actions/setup-kairous
      - name: Next.js build cache を復元
        uses: actions/cache@<cache-sha>
        with:
          path: .next/cache
          key: nextjs-${{ hashFiles('bun.lockb') }}-${{ hashFiles('src/**', 'public/**', 'next.config.ts', 'tsconfig.json') }}
          restore-keys: |
            nextjs-${{ hashFiles('bun.lockb') }}-
      - name: Next.js 本番ビルド
        run: bun run build
      - name: ビルド成果物を artifact にアップロード
        uses: actions/upload-artifact@65c4c4a1ddee5b72f698fdd19549f0f0fb45cf08 # v4.6.0
        with:
          name: nextjs-build
          path: |
            .next
            !.next/cache
          retention-days: 1
          if-no-files-found: error
```

環境変数 `NEXT_PUBLIC_SUPABASE_*` は build に必要。Kairous は `env.ts` で validate するため、ビルド時に環境変数が必須。`bun run build` の前に設定する:

```yaml
      - name: ビルド用ダミー環境変数
        run: |
          # build 時の env.ts バリデーションを通過するためのダミー値。
          # 実行時 (lighthouse/test-large) では各ジョブで本物の Supabase ローカルを起動し上書きする
          echo "NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321" >> $GITHUB_ENV
          echo "NEXT_PUBLIC_SUPABASE_ANON_KEY=dummy-anon-key-for-build-only" >> $GITHUB_ENV
          echo "SUPABASE_SERVICE_ROLE_KEY=dummy-service-role-key-for-build-only" >> $GITHUB_ENV
```

このステップを `Next.js 本番ビルド` の **直前** に挿入。

- [ ] **Step 3: src/lib/env.ts の確認**

`src/lib/env.ts` を読み、build 時に環境変数不足でエラーになる validation があるか確認。NEXT_PUBLIC_ 系は client bundle に埋め込まれるため build 時に必須。ダミー値が許容されるか (形式チェックに URL parse があるかなど) を確認する。

もし URL 形式や length 制約があればダミー値を `http://127.0.0.1:54321` / 適切な長さの文字列に調整する。

## Task 4: lighthouse ジョブで build artifact を消費

**Files:**
- Modify: `.github/workflows/lighthouse-ci.yml`

- [ ] **Step 1: actions/download-artifact の SHA を解決**

```bash
gh api repos/actions/download-artifact/git/refs/tags/v4.1.8 --jq .object.sha
```

- [ ] **Step 2: lighthouse ジョブを書き換え**

lighthouse-ci.yml lighthouse ジョブから以下 2 ステップを削除:

```yaml
      - name: Next.js 本番ビルド
        run: bun run build
```

代わりに「環境変数を設定」ステップの後、「Next.js サーバ起動 (バックグラウンド)」の前に挿入:

```yaml
      - name: ビルド成果物を download
        uses: actions/download-artifact@<download-sha>
        with:
          name: nextjs-build
          path: .next
```

lighthouse-ci.yml は別ワークフローのため、ci.yml の build ジョブとは別 run ID になる。artifact の受け渡しは **同一 workflow run 内でしか動作しない** ため、lighthouse-ci.yml を ci.yml に統合するか、build ジョブを lighthouse-ci.yml にも配置する必要がある。

**方針: build ジョブを lighthouse-ci.yml と ci.yml の両方に置くのは重複になるため、lighthouse-ci.yml を ci.yml に統合する (jobs.lighthouse として)。**

- [ ] **Step 3: lighthouse-ci.yml を ci.yml に統合**

`.github/workflows/lighthouse-ci.yml` の `jobs.lighthouse` 定義を `.github/workflows/ci.yml` の `jobs` 配下にコピーする。依存関係は `needs: [build]` に設定。元の `.github/workflows/lighthouse-ci.yml` は削除する。

paths-ignore とトリガー条件は ci.yml が既に同等 (pull_request + main への push) をカバーしているため追加不要。

統合後の lighthouse ジョブ (ci.yml 内):

```yaml
  lighthouse:
    runs-on: ubuntu-latest
    needs: [build]
    steps:
      - uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
      - uses: ./.github/actions/setup-kairous
      - uses: supabase/setup-cli@b60b5899c73b63a2d2d651b1e90db8d4c9392f51 # v1.6.0
        with:
          version: "2.76.7"
      - name: Playwright ブラウザインストール
        run: bunx playwright install --with-deps chromium
      - name: Supabase ローカル起動
        run: supabase start -x realtime,storage,imgproxy,pgadmin-schema-diff,migra,studio,logflare,vector,supavisor
      - name: 環境変数を設定
        run: |
          eval $(supabase status -o env | grep -E '^(API_URL|ANON_KEY|SERVICE_ROLE_KEY)=')
          echo "NEXT_PUBLIC_SUPABASE_URL=${API_URL}" >> $GITHUB_ENV
          echo "NEXT_PUBLIC_SUPABASE_ANON_KEY=${ANON_KEY}" >> $GITHUB_ENV
          echo "SUPABASE_SERVICE_ROLE_KEY=${SERVICE_ROLE_KEY}" >> $GITHUB_ENV
      - name: ビルド成果物を download
        uses: actions/download-artifact@<download-sha>
        with:
          name: nextjs-build
          path: .next
      - name: Next.js サーバ起動 (バックグラウンド)
        run: |
          bun run start &
          bunx wait-on http://localhost:3000 --timeout 60000
      - name: 認証 storageState を生成
        env:
          LHCI_REUSE_SERVER: "1"
          LHCI_SKIP_TEARDOWN: "1"
        run: bun run lhci:setup
      - name: Lighthouse 計測用の Chrome パスを明示
        run: |
          CHROME_PATH=$(which google-chrome-stable)
          if [ -z "$CHROME_PATH" ]; then
            echo "google-chrome-stable が見つからない" >&2
            exit 1
          fi
          echo "CHROME_PATH=${CHROME_PATH}" >> $GITHUB_ENV
      - name: Lighthouse CI 実行
        env:
          PUPPETEER_EXECUTABLE_PATH: ${{ env.CHROME_PATH }}
        run: bun run lhci
      - name: Lighthouse レポートを artifact に保存
        if: ${{ !cancelled() }}
        uses: actions/upload-artifact@65c4c4a1ddee5b72f698fdd19549f0f0fb45cf08 # v4.6.0
        with:
          name: lighthouse-report
          path: .lighthouseci/
          retention-days: 7
```

**重要**: build ジョブで環境変数にダミー値を設定した場合、build 時点での `NEXT_PUBLIC_SUPABASE_URL` は `127.0.0.1:54321` として bundle に焼き込まれる可能性がある。Supabase ローカルも同じポートを使うので結果的に一致するが、ANON_KEY が異なる場合は client bundle 内の値が実サーバと不整合になる。

Kairous では `NEXT_PUBLIC_SUPABASE_ANON_KEY` は browser 側の Supabase client 初期化に使われる。ダミー ANON_KEY が bundle に焼き込まれると Lighthouse 計測時の page 読み込みで認証が壊れる可能性がある。

**対処**: Lighthouse ジョブでは build artifact を使っているが、実行時の env var (GITHUB_ENV で設定されたもの) は **build bundle 内に焼き込まれた NEXT_PUBLIC_** を上書きできない。これは Next.js の仕様。

**解決策の決定**: build 時点で本物の Supabase ローカルを起動して正しい ANON_KEY で build する。つまり build ジョブに Supabase 起動ステップを含める。これで:
- build 時点の NEXT_PUBLIC_ANON_KEY = Supabase ローカル ANON_KEY
- build artifact が lighthouse / test-large で使われる時、それぞれのジョブで同じ Supabase バージョンのローカルを起動すれば ANON_KEY は同一 (Supabase CLI の固定 JWT を出力するため)

- [ ] **Step 4: build ジョブを修正 (Supabase ローカル起動を追加)**

Task 3 で作成した build ジョブを書き直す。「ビルド用ダミー環境変数」ステップを削除し、代わりに:

```yaml
      - uses: supabase/setup-cli@b60b5899c73b63a2d2d651b1e90db8d4c9392f51 # v1.6.0
        with:
          version: "2.76.7"
      - name: Supabase ローカル起動
        run: supabase start -x realtime,storage,imgproxy,inbucket,pgadmin-schema-diff,migra,studio,edge-runtime,logflare,vector,supavisor
      - name: 環境変数を設定
        run: |
          eval $(supabase status -o env | grep -E '^(API_URL|ANON_KEY|SERVICE_ROLE_KEY)=')
          echo "NEXT_PUBLIC_SUPABASE_URL=${API_URL}" >> $GITHUB_ENV
          echo "NEXT_PUBLIC_SUPABASE_ANON_KEY=${ANON_KEY}" >> $GITHUB_ENV
          echo "SUPABASE_SERVICE_ROLE_KEY=${SERVICE_ROLE_KEY}" >> $GITHUB_ENV
```

これを「Next.js build cache を復元」の前 / `Next.js 本番ビルド` の前に配置する。

Supabase CLI の JWT は deterministic (anon / service_role key は version 固定なら同一) であることを確認する。`supabase/setup-cli` バージョン `2.76.7` で固定しているため build / lighthouse / test-large の ANON_KEY は一致する。

この検証は実装時に実行ログで確認: 3 ジョブの ANON_KEY が一致すればよい。

## Task 5: test-large を matrix shard 化 + artifact 消費

- [ ] **Step 1: test-large ジョブを matrix 化**

```yaml
  test-large:
    runs-on: ubuntu-latest
    needs: [test-medium, lint-and-typecheck, build]
    strategy:
      fail-fast: false
      matrix:
        shard: [1, 2, 3, 4]
    steps:
      - uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd # v6.0.2
      - uses: ./.github/actions/setup-kairous
      - uses: supabase/setup-cli@b60b5899c73b63a2d2d651b1e90db8d4c9392f51 # v1.6.0
        with:
          version: "2.76.7"
      - name: Playwright ブラウザインストール
        run: bunx playwright install --with-deps chromium
      - name: Supabase ローカル起動 (Inbucket 含む)
        run: supabase start -x realtime,storage,imgproxy,pgadmin-schema-diff,migra,studio,logflare,vector,supavisor
      - name: 環境変数を設定
        run: |
          eval $(supabase status -o env | grep -E '^(API_URL|ANON_KEY|SERVICE_ROLE_KEY)=')
          echo "NEXT_PUBLIC_SUPABASE_URL=${API_URL}" >> $GITHUB_ENV
          echo "NEXT_PUBLIC_SUPABASE_ANON_KEY=${ANON_KEY}" >> $GITHUB_ENV
          echo "SUPABASE_SERVICE_ROLE_KEY=${SERVICE_ROLE_KEY}" >> $GITHUB_ENV
      - name: ビルド成果物を download
        uses: actions/download-artifact@<download-sha>
        with:
          name: nextjs-build
          path: .next
      - name: E2E テスト (shard ${{ matrix.shard }}/4)
        run: bunx playwright test --config tests/large/playwright.config.ts --shard=${{ matrix.shard }}/4
      - uses: actions/upload-artifact@65c4c4a1ddee5b72f698fdd19549f0f0fb45cf08 # v4.6.0
        if: ${{ !cancelled() }}
        with:
          name: playwright-report-shard-${{ matrix.shard }}
          path: |
            playwright-report/
            test-results/
          retention-days: 7
```

      timeout: 60_000,

  : {
      command: "bun run build && bun run start",
      url: "http://localhost:3000",

## Task 6: 計測と PR description 更新

**Files:** 変更なし (PR description の更新のみ)

- [ ] **Step 1: baseline 計測**

```bash
gh run list --workflow ci.yml --branch main --limit 5 --json databaseId,conclusion,createdAt,updatedAt | \
  jq 'map(select(.conclusion == "success") | {id: .databaseId, durationMin: ((.updatedAt | fromdateiso8601) - (.createdAt | fromdateiso8601)) / 60})'
```

同様に lighthouse-ci.yml も取得 (統合前のデータ)。中央値を算出。ジョブ別の所要時間は `gh run view <run-id> --json jobs` で各 job の `startedAt` / `completedAt` 差分から計算。

集計する指標:
- ci.yml 全体
- build ジョブ
- test-large (shard 別 + 最長)
- lighthouse ジョブ
